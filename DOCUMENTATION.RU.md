## Что это за программа

**XTTS Studio** — портативное Windows-приложение для локальной озвучки и клонирования голоса на базе **XTTS v2**.

Основной сценарий простой:

```text
короткий образец голоса + текст → готовый WAV или MP3
```

Что работает локально после установки:

- XTTS-генерация;
- обработка референса;
- нормализация и разбиение текста;
- очередь задач и история;
- локальная RVC-конвертация, если модель уже скачана;
- локальные GGUF-модели через `llama-cpp-python`.

Интернет нужен не всегда. Он используется для облачного AI, поиска и скачивания RVC-моделей, загрузки локальных LLM из каталога, проверки обновлений и установки некоторых компонентов.

> **Важно:** «офлайн» относится к самой генерации и локальным моделям. Поиск в сетевом каталоге и облачные API по определению требуют подключения.

## Быстрый старт

1. Распакуйте XTTS Studio в каталог без кириллицы.
2. Запустите `XTTS Studio.exe`.
3. Выберите референс голоса длительностью примерно **10–20 секунд**.
4. Введите текст.
5. Выберите пресет.
6. Нажмите **🚀 ГЕНЕРИРОВАТЬ**.
7. Готовый файл появится в `outputs/` и в окне **🎵 Аудио**.

```text
✔ C:\XTTS\
✔ D:\Apps\XTTS-Studio\
✘ C:\Новая папка\XTTS\
```

### Что такое `XTTS Studio.exe`

`XTTS Studio.exe` — не отдельная копия движка и не монолитная сборка программы. Это небольшой лаунчер, сделанный из стартового BAT-файла: внутри находятся только пути запуска bundled Python, ссылка на `gui.py` и иконка приложения.

Лаунчер включается в обновления, потому что при изменении структуры portable-сборки может понадобиться переписать пути к runtime или окружению. Модели, настройки и пользовательские данные внутри `.exe` не хранятся.

### Как обновляется релизная версия

Полный архив нужен для первой установки или редкого несовместимого обновления. Обычные новые версии устанавливаются встроенным updater на стороне клиента:

1. приложение получает список изменённых файлов;
2. скачивает только нужные файлы во временный staging;
3. проверяет SHA256, если хэш есть в manifest;
4. делает backup заменяемых файлов;
5. применяет обновление и ждёт подтверждения успешного запуска;
6. при неудачном старте может выполнить rollback.

Поэтому для каждого обычного релиза не требуется заново скачивать и распаковывать многогигабайтную portable-сборку. Если текущая версия ниже `min_app_version` или структура сборки изменилась несовместимо, программа попросит выполнить полную переустановку.

`XTTS Studio.exe` также входит в manifest обновления: если меняются пути запуска, updater заменит лаунчер вместе с остальными изменёнными файлами.

### Первый запуск на CPU

Приложение сразу может работать на CPU. Это самый совместимый режим, но длинные тексты и RVC будут обрабатываться медленнее.

### Включение NVIDIA CUDA

Если установлена подходящая NVIDIA GPU:

```text
⚙ Настройки → Ускорение → установить GPU-вариант PyTorch
```

Не копируйте системный Python и не устанавливайте зависимости глобально: приложение использует собственное окружение `python/xtts_env`.

## Как подготовить хороший референс

Референс влияет на результат сильнее большинства ползунков.

Хороший образец:

- один человек;
- 10–20 секунд связной речи;
- без музыки и посторонних голосов;
- без сильного эха, реверберации и шумоподавления;
- без клиппинга;
- с тем же языком и близкой манерой речи, которую вы хотите получить.

Плохой пример: тихая речь на фоне музыки в большой комнате.  
Хороший пример: обычная речь на расстоянии 15–30 см от микрофона в тихой комнате.

### Что делает программа с референсом

`engine/reference_processor.py`:

1. создаёт папку голоса в `library/`;
2. переводит файл в mono WAV 24 кГц;
3. аккуратно обрезает тишину по краям;
4. ограничивает обрабатываемый фрагмент 30 секундами;
5. применяет компрессию и нормализацию примерно к −16 dBFS;
6. сохраняет `normalized.wav`;
7. оценивает сигнал/шум (SNR).

Оценка SNR:

| SNR | Оценка | Что ожидать |
|-----|--------|--------------|
| ≥25 dB | excellent | чистый и стабильный референс |
| 15–25 dB | good | обычно достаточно |
| 8–15 dB | poor | возможны шум, нестабильный тембр и дыхание |
| <8 dB | bad | лучше перезаписать образец |

Повторно выбранный `library/<voice>/normalized.wav` не нормализуется ещё раз: это защищает запись от накопительной компрессии.

## Практические сценарии

### 1. Глава аудиокниги

1. Выберите спокойный чистый референс.
2. Используйте пресет **Нарратив**.
3. Вставьте главу целиком.
4. Оставьте QC включённым.
5. После генерации прослушайте спорные места в **Истории** по wave-форме.
6. Верните текст нужной записи в редактор кнопкой **↩** и перегенерируйте только проблемный фрагмент.

Программа сама разобьёт текст на чанки и затем склеит их с паузами. Жёсткого лимита на размер текста в интерфейсе нет, но время обработки, RAM и размер кэша растут вместе с объёмом.

### 2. Голос персонажа через RVC

1. Сначала добейтесь разборчивого XTTS без RVC.
2. Включите RVC и выберите скачанную `.pth`-модель.
3. Проверьте модель отдельной кнопкой параметрического preview.
4. Подберите Pitch shift.
5. Меняйте Index только если рядом есть подходящий `.index`.
6. Сравните результат с отключённым RVC, а не только с demo на сайте.

Если исходный XTTS говорит неразборчиво, RVC обычно не исправит дикцию: он меняет тембр уже созданного звука.

### 3. Несколько файлов TXT

Для набора глав, реплик или рекламных вариантов используйте пакетную обработку. Каждая задача попадёт в очередь с собственным текстом и выбранными параметрами. Задачу можно отменить до старта или во время выполнения.

### 4. Русский текст с числами и сокращениями

Пример:

```text
В 2026 году А. С. Пушкин не мог пользоваться GPU, API и OpenAI.
```

Перед XTTS текст проходит нормализацию, словарь произношений и безопасное разбиение. Инициалы вроде `А. С. Пушкин` не должны разрываться как отдельные предложения.

### 5. Озвучка с AI Conductor

Conductor полезен, когда длинный текст содержит разные сцены: спокойное вступление, диалог и эмоциональный финал. Он может назначить разные параметры и паузы для чанков. Для ровного технического текста обычный пресет часто предсказуемее.

## Как проходит генерация

```text
референс
  → конвертация / нормализация / SNR
  → speaker embedding и его кэш

текст
  → optional AI preprocessing
  → normalizer
  → словарь произношений
  → chunker
  → optional AI Conductor / prosody / smart pauses

каждый чанк
  → XTTS
  → QC и при необходимости повторная попытка
  → optional RVC
  → кэш чанка

финал
  → склейка с паузами
  → нормализация громкости
  → optional de-esser
  → WAV или MP3
```

### Что такое чанк

Чанк — небольшой фрагмент текста, который XTTS генерирует отдельно. `TextChunker` ориентируется примерно на:

```text
минимум: 50 символов
цель:    150 символов
максимум: 175 символов
```

Разрез выбирается по знакам препинания. Chunker старается не оставлять в начале или конце отдельные союзы вроде «и», «но», «который».

### QC

При включённом `qc_enabled` неудачный чанк может быть сгенерирован повторно до трёх раз. Проверяются в том числе подозрительные повторы и неправдоподобная длительность.

### Кэш чанков

Одинаковый текст с одинаковым голосом и параметрами можно не генерировать заново. Ключ кэша учитывает RVC-настройки, поэтому версия без RVC не смешивается с версией после RVC.

## Пресеты и параметры качества

В программе четыре пресета:

| Пресет | Когда использовать |
|--------|---------------------|
| **Высокое качество** | универсальный стартовый вариант |
| **Нарратив** | книги, лекции, спокойный закадр |
| **Динамика** | ролики, короткая реклама, энергичная подача |
| **Экспрессия** | эмоциональные реплики и сцены; требует подходящего референса |

У каждого пресета собственные значения. Изменение «Нарратива» не обязано менять «Динамику».

Вкладки настроек:

- **RVC** — модель, Index, Pitch shift, f0;
- **Обрезка** — режим и длина trim;
- **Вывод** — WAV/MP3, QC, de-esser;
- **XTTS** — temperature, top_p, top_k, repetition penalty, speed, prosody.

Настройки сохраняются в `settings.json` внутри `quality_params`. Запись выполняется через read-modify-write, поэтому несвязанные ключи темы и интерфейса не должны исчезать.

### Как настраивать без хаоса

1. Начните с пресета по умолчанию.
2. Меняйте один параметр за раз.
3. Сравнивайте на одном и том же коротком тексте.
4. Не оценивайте тембр по одной фразе: проверьте вопрос, утверждение и длинное предложение.
5. Сохраните удачный вариант как отдельный пресет темы/раскладки только если речь именно о UI; голосовые параметры хранятся в quality preset.

## RVC: выбор модели, preview и кэш

RVC — второй этап обработки: он получает готовый WAV от XTTS и меняет его тембр. В проекте RVC запускается **для каждого принятого QC-чанка**. Если RVC завершается ошибкой, задача не падает целиком: сохраняется исходный XTTS-чанк.

### Файлы модели

```text
models/rvc/<имя>.pth       обязательные веса
models/rvc/<имя>.index     необязательный индекс признаков
```

- `.pth` нужен всегда;
- `.index` улучшает сходство для моделей, обученных с индексом;
- ползунок Index не может дать эффект без подходящего `.index`;
- модель неизвестного происхождения может быть низкого качества независимо от числа эпох.

### Каталоги моделей

В браузере доступны:

| Раздел | Откуда данные |
|--------|---------------|
| **★ Подборка** | локальный seed и `catalog_cache.json`; работает офлайн |
| **🆕 Новые** | первая страница публичного каталога voice-models.com |
| **🔥 Топ** | публичный раздел `/top` |
| Поиск | локальные записи + live-поиск voice-models.com |

Сетевой каталог — удобный индекс, а не гарантия качества или лицензии модели.

Действия строки:

- **▶ / ■** — проиграть/остановить demo;
- **⬇** — скачать прямой `.pth` или `.zip`;
- **✕** — отменить загрузку;
- **🔗** — открыть страницу, если прямого файла нет;
- **🗑** — удалить локальную модель и связанный `.index`.

### Два разных preview

**Demo из каталога** показывает готовый пример автора модели. Он помогает понять общий тембр, но не показывает, как модель обработает именно ваш референс.

**Параметрический preview** рядом с выбором модели выполняет реальный локальный RVC-проход:

- источник — первые 6 секунд текущего voice reference;
- используются текущие Index, Pitch shift и f0 method;
- расчёт идёт в фоне;
- одинаковая комбинация берётся из кэша;
- на модель хранится до шести последних вариантов;
- если параметры изменились во время расчёта, устаревший результат не запускается автоматически.

Это наиболее честный способ сравнивать параметры на своём голосе.

### Практическая настройка

- **Pitch shift**: меняйте по одному полутону, пока высота не станет естественной. `+12` и `−12` — сдвиг на октаву, а не универсальная настройка пола голоса.
- **Index**: начните около середины диапазона. Слишком высокое значение может усилить артефакты; без `.index` оно не поможет.
- **f0 method**: `rmvpe` — разумная отправная точка. Альтернативы полезны, если конкретная модель плохо отслеживает высоту исходного голоса.

### Кэш и метаданные

| Путь | Что хранится |
|------|--------------|
| `.preview_cache/` | короткие demo с сайта |
| `.parameter_preview_cache/<model>/` | WAV-варианты вашего parameter preview |
| `.metadata/<model>.json` | страница, URL, автор и путь к preview локальной модели |
| `catalog_cache.json` | кэш каталога |

Кнопка **🧹** вызывает `clear_rvc_cache()`: удаляет orphan-preview, временные файлы и остатки прерванных загрузок. Она не удаляет установленные `.pth`, `.index` и preview, закреплённые за установленной моделью. Связанный кэш удаляется вместе с моделью.

### Если локальная модель без кнопки ▶

Сам `.pth` не содержит demo-аудио. Для ранее скачанной модели программа пытается восстановить метаданные из seed, New/Top и текущего поиска. Если совпадение не найдено, модель всё равно работает для конвертации, но preview недоступен.

## Обработка текста

| Этап | Что делает | Пример |
|------|------------|--------|
| `normalizer.py` | числа, даты, сокращения, допустимые символы | `2026` → слова |
| `word_replacer.py` | пользовательские и автоматически найденные произношения | `GPU`, имя бренда, иностранный термин |
| `chunker.py` | делит длинный текст на безопасные части | не режет `А. С. Пушкин` как три предложения |
| `prosody_layer.py` | добавляет смысловые паузы и акценты | перечисление или вывод |
| `smart_pauses.py` | рассчитывает паузы при склейке | точка длиннее запятой |

Приоритет правил словаря:

```text
builtin → auto → ai_corrected → custom
```

Пользовательское правило имеет наивысший приоритет. Перед изменениями создаются резервные копии; хранится до 30.

Когда активен AI Conductor, обычные prosody/smart pauses пропускаются: паузы приходят из `conductor_map`.

## Очередь, история и пакетная обработка

### Очередь

`TaskManager` выполняет задачи последовательно в worker-потоке. Отмена текущей задачи выставляет флаг, который проверяется внутри `run_tts`. Задача, ещё не начавшаяся, помечается отменённой и не запускается.

При закрытии приложения manager:

- отменяет текущую задачу;
- вычищает ожидающие;
- будит заблокированный `queue.get()` служебным sentinel;
- ждёт завершения worker с таймаутом.

### История

В `history.json` сохраняются последние **100** генераций: дата, текст, голос, пресет, путь, длительность и число чанков.

В окне истории:

- wave-форма находится прямо на карточке;
- клик по волне перематывает;
- **▶ / ■** запускает и останавливает;
- громкость находится рядом;
- **↩** возвращает текст в редактор.

### Пакетная обработка

Batch-режим подходит для нескольких TXT-файлов. Он использует ту же очередь и те же quality parameters, что обычная генерация.

## AI-функции и работа без интернета

### Что может обращаться в сеть

| Действие | Интернет |
|----------|----------|
| XTTS с локальным референсом | не нужен |
| RVC с уже установленной моделью | не нужен |
| Каталог/поиск/скачивание RVC | нужен |
| Облачный AI-провайдер | нужен |
| Локальная GGUF-модель после скачивания | не нужен |
| Проверка обновлений / установка CUDA-компонентов | нужен |

### AI preprocessing

Если включён AI edit, исходный текст сначала отправляется выбранному AI-провайдеру, а затем проходит локальный normalizer. При недоступности AI функция должна вернуть исходный текст, а не остановить TTS.

### AI Conductor

Conductor может:

- назначать параметры по чанкам;
- задавать `pause_after_ms`;
- добавлять исправления произношения;
- при включённом rewrite менять стиль текста.

Если важна полная приватность, используйте локальную GGUF-модель или выключите AI-функции.

### Настройки API

Ключи хранятся в `gpt_settings.json`. Провайдеры могут образовывать fallback-цепочку: активный → другие настроенные built-in → custom. Никогда не публикуйте `gpt_settings.json` вместе с реальными ключами.

## CPU, CUDA и скорость

| Режим | Плюсы | Ограничения |
|-------|-------|-------------|
| CPU | работает почти везде, не требует NVIDIA | генерация, RVC и local LLM заметно медленнее |
| CUDA | быстрее XTTS и часть локальных задач | нужна совместимая NVIDIA GPU и правильный PyTorch |

Практические советы:

- сначала проверьте короткую фразу;
- только потом запускайте главу или batch;
- RVC добавляет отдельный этап на каждый чанк;
- parameter preview на CPU может рассчитываться несколько секунд;
- отсутствие движения progress bar не всегда означает зависание: загрузка модели и первый inference самые долгие;
- не устанавливайте другую версию torch поверх portable-окружения вручную.

## Системное окружение и восстановление

XTTS Studio не использует системный Python. Все управляемые пакеты устанавливаются в:

```text
python/xtts_env/Lib/site-packages
```

Временные файлы сборки и общий pip-кэш находятся в `python/temp` и `python/pip_cache`. Torch, RVC и llama.cpp используют эти каталоги совместно, поэтому одновременно запускать несколько установок нельзя.

Публичная точка доступа — `engine/env_core/__init__.py`. Она реэкспортирует hardware detection, установщики Torch/llama.cpp/RVC, диагностику, очистку и recovery.

| Зона | Основной публичный API |
|------|------------------------|
| Hardware | `detect_cpu`, `detect_gpu`, `PYTHON_EXE`, `PROJECT_ROOT` |
| Torch | `install_torch`, `uninstall_torch`, `torch_status`, `cancel_install_torch`, checkpoints и broken variants |
| llama.cpp | `install_llama_cpp`, `uninstall_llama_cpp`, `llama_cpp_status`, `resolve_backend`, startup state |
| RVC | `install_rvc`, `uninstall_rvc`, `rvc_status` |
| Diagnostics | `run_full_diagnostics`, `scan_for_garbage`, `finalize_deletion`, `run_error_recovery`, cache helpers |

### Определение CPU и GPU

`engine/env_core/cpu_gpu.py` собирает сведения без импорта тяжёлых ML-библиотек.

**CPU**

- использует `py-cpuinfo`;
- если пакета нет, может быстро установить его во встроенный `site-packages`;
- определяет имя CPU и флаги AVX, AVX2, FMA/FMA3, F16C;
- эти флаги используются при CPU-сборке `llama-cpp-python`, чтобы отключить инструкции, которых нет у конкретного процессора.

**GPU**

- NVIDIA определяется через `nvidia-smi`;
- считываются имя, reported CUDA version и VRAM;
- AMD/Intel определяются через PowerShell/WMI;
- для AMD/Intel VRAM дополнительно читается из 64-битного значения реестра, потому что `Win32_VideoController.AdapterRAM` ограничен 32 битами.

Наличие AMD/Intel GPU не означает поддержку CUDA. Для Torch в этом проекте GPU-вариант — NVIDIA CUDA; локальный llama.cpp может отдельно выбрать Vulkan для AMD/Intel.

### Установка Torch

`engine/env_core/torch_setup.py` управляет согласованным набором:

```text
torch       2.2.2
torchaudio  2.2.2
torchvision 0.17.2
```

Доступны варианты:

- `cu118` — NVIDIA CUDA 11.8;
- `cpu` — универсальный CPU build.

Как выбирается вариант:

1. учитывается сохранённое предпочтение CPU/GPU;
2. проверяется NVIDIA и reported CUDA не ниже 11.8;
3. исключаются варианты, ранее помеченные как broken;
4. запрос CUDA на не-NVIDIA машине принудительно переводится в CPU.

Установка:

- защищена общим install-lock, поэтому два `pip install` не запускаются одновременно;
- сохраняет этап в `.torch_install_checkpoint.json` и умеет продолжить прерванный процесс;
- очищает прежние `torch*`, `functorch`, `triton`, `nvidia_*` из target-папки;
- использует локальный `python/pip_cache`;
- проверяет импорт в отдельном процессе;
- для CUDA дополнительно требует `torch.cuda.is_available() == True`;
- при неудаче помечает вариант broken и переходит на CPU;
- сохраняет успешный выбор в `.torch_installed_variant.json`.

`cancel_install_torch()` завершает активный pip-процесс и освобождает install-lock. `clean_torch_cache()` удаляет общий temp/pip cache и checkpoint.

**Практический случай:** CUDA wheel установился, но драйвер или GPU не позволяют `torch.cuda.is_available()`. Программа не должна оставлять такой вариант активным: `cu118` помечается в `.torch_broken_variants.json`, затем устанавливается CPU build.

### Установка локального llama.cpp

`engine/env_core/llama_setup.py` выбирает backend отдельно от Torch:

| Устройство | Предпочтительный backend |
|------------|-------------------------|
| NVIDIA + CUDA | CUDA prebuilt wheel |
| AMD / Intel | Vulkan prebuilt wheel |
| нет подходящего GPU или backend помечен broken | CPU build |

CPU-вариант может собираться из исходников. `build_cmake_args()` отключает AVX/AVX2/FMA/F16C, которых нет у обнаруженного CPU, а сборка использует доступные ядра.

`llama_cpp_status()` сначала проверяет целостность пакета (`__init__.py`, `llama.py`, `llama_cache.py`), затем импортирует его в отдельном процессе. Неполная папка трактуется как прерванная установка, а не как рабочий пакет.

После установки выполняются две проверки:

1. импорт `llama_cpp` в отдельном процессе;
2. `smoke_test_gpu_init()` для GPU — реальное создание `Llama(..., n_gpu_layers=-1)` на найденной GGUF-модели, если она доступна.

Если CUDA/Vulkan wheel импортируется, но падает при реальной инициализации модели, backend записывается в `.llama_broken_backends.json`, после чего установщик переходит на CPU. Это важнее простой проверки `import llama_cpp`.

Состояние незавершённой установки хранится в `.llama_install_checkpoint.json`; выбранный backend — в `.llama_installed_backend.json`. Если библиотека уже работает, забытый checkpoint очищается автоматически.

Во время долгой CPU-сборки watchdog проверяет активность файлов. Сообщение «файлы давно не менялись» — предупреждение о возможном зависании, а не автоматическая отмена.

### Установка RVC-окружения

`engine/env_core/rvc_setup.py` устанавливает `rvc-python` и его зависимости в то же portable-окружение.

Почему это сложнее обычного `pip install`:

- `fairseq` на Windows часто требует компиляции;
- metadata старых зависимостей содержит несовместимые жёсткие pins;
- `pip --target` не всегда считает уже лежащий в target Torch установленным;
- повторная установка могла бы зря скачать CUDA Torch на несколько гигабайт;
- запущенное приложение может временно держать `.pyd`/DLL.

Что делает установщик:

- определяет уже установленный Torch build (`cpu` или `cu118`) и использует тот же индекс;
- строит динамический constraint по версиям, реально лежащим в `site-packages`;
- ставит `rvc-python --no-deps`, затем читает реальные `Requires-Dist` из его METADATA;
- на Windows использует готовые fairseq wheels для Python 3.10/3.11/3.12, когда они доступны;
- не устанавливает устаревший backport `dataclasses` поверх Python 3.11;
- отдельно доставляет зависимости fairseq/sacrebleu, сохраняя общий `portalocker`;
- принудительно восстанавливает совместимые NumPy и PyYAML;
- при WinError 5 повторяет установку без `--upgrade`, чтобы не перезаписывать залоченные файлы;
- до шести раз доустанавливает найденные отсутствующие модули;
- в конце проверяет `from rvc_python.infer import RVCInference` в отдельном процессе.

`uninstall_rvc()` удаляет RVC/fairseq-специфичные пакеты, но намеренно оставляет `portalocker`, потому что это общая зависимость, используемая и другими частями окружения.

**Практический случай:** RVC устанавливается в CPU-сборку. Динамический constraint закрепляет уже установленный `torch+cpu`, поэтому pip не пытается заменить его на `cu118` и повторно скачать несколько гигабайт.

### Полная диагностика

`run_full_diagnostics()` проверяет 11 компонентов в **одном изолированном subprocess**:

```text
critical:
  numpy, torch, torchaudio, torchvision, TTS,
  soundfile, pygame, customtkinter, num2words

optional:
  llama_cpp, rvc_python
```

Optional-компонент может быть просто не установлен — это нормальное состояние и не блокирует запуск основного TTS. `get_optional_status()` различает `ok`, `not_installed` и `broken`; предупреждать как о поломке следует только в последнем случае.

Изоляция нужна не только для надёжности результата. Если импортировать Torch/TTS прямо в GUI-процесс, Windows может удерживать `.pyd` и DLL, после чего recovery не сможет заменить их. Subprocess завершает работу и освобождает файлы.

Если NumPy не импортируется, зависимые проверки Torch/TTS помечаются `SKIPPED`: сначала нужно восстановить первопричину, а не переустанавливать всё подряд.

Успешный полный результат кэшируется в `.env_diagnostics_cache.json`. Кэш используется только пока совпадают:

- путь к Python;
- mtime `site-packages`;
- число элементов в `site-packages`.

Перед проверкой удаляется случайно установленный backport `dataclasses`, который на Python 3.11 может перекрыть stdlib и сломать Torch, torchvision и сам pip.

`clear_diagnostics_cache()` заставляет следующую проверку пройти заново. `clean_pip_download_cache()` удаляет общий `python/temp` и `python/pip_cache` сразу для Torch, RVC и llama.cpp; вызывать её во время активной установки нельзя.

### Сканирование мусора и карантин

`scan_for_garbage(mode="fast" | "deep")` не удаляет найденные файлы сразу.

Порядок:

1. запускается baseline-диагностика;
2. сканируются `python/temp`, `python/pip_cache`, `logs`, `__pycache__`, `.pytest_cache` и известные временные расширения;
3. `models`, `outputs`, `library`, `reference`, `.git` исключаются из общего обхода;
4. кандидаты перемещаются в `python/xtts_env/Quarantine`;
5. диагностика запускается повторно;
6. если появился новый сбой, файлы автоматически возвращаются и помечаются unsafe;
7. только подтверждённый список окончательно удаляется через `finalize_deletion()`.

**Fast** использует уже известный кэш безопасных файлов. Новые неизвестные файлы пропускаются.  
**Deep** проверяет весь набор кандидатов через карантин и повторную диагностику.

История safe/unsafe/deleted хранится в `.known_safe_files.json` и используется при последующем восстановлении.

### Автовосстановление зависимостей

`run_error_recovery()` сопоставляет историю удалений с пакетами и версиями из `requirements.txt`, но перед установкой проверяет реальное состояние окружения. Рабочий пакет не переустанавливается только потому, что в истории осталась старая запись.

Особые случаи:

- RVC восстанавливается через специализированный `install_rvc()`, а не обычный pip;
- Torch-вариант выбирается по фактически установленной metadata и сохранённому backend;
- рабочий Torch не скачивается повторно при ремонте SoundFile или torchvision;
- несовместимый PyAV (`av.logging`) чинится до/после torchvision;
- результат pip подтверждается повторным импортом: код возврата 0 сам по себе не считается доказательством;
- в начале и конце снова удаляется теневой `dataclasses` backport;
- подробный вывод пишется в `logs/recovery_pip_output.log`.

**Практический случай:** сканирование удалило файл, связанный с `av`, и torchvision перестал импортироваться. Recovery сначала восстанавливает совместимый PyAV и проверяет `av.logging`; если torchvision после этого снова работает, его wheel не скачивается заново.

### Важные служебные файлы

| Путь | Назначение |
|------|------------|
| `.env_diagnostics_cache.json` | кэш успешной полной диагностики |
| `.known_safe_files.json` | safe/unsafe/deleted history сканера |
| `.torch_install_checkpoint.json` | этап незавершённой установки Torch |
| `.torch_installed_variant.json` | последний успешный `cpu` / `cu118` |
| `.torch_broken_variants.json` | варианты Torch, которые не следует выбирать автоматически |
| `.llama_install_checkpoint.json` | этап установки llama.cpp |
| `.llama_installed_backend.json` | последний успешный backend llama.cpp |
| `.llama_broken_backends.json` | CUDA/Vulkan backends, не прошедшие проверку |
| `python/temp/` | временные файлы pip/build |
| `python/pip_cache/` | общий кэш загрузок установщиков |

## Частые проблемы

### Программа не запускается из новой папки

Проверьте путь. Кириллица и некоторые нестандартные символы ломают часть portable-зависимостей.

### Голос похож слабо

1. Отключите RVC.
2. Проверьте XTTS на чистом референсе.
3. Уберите музыку, эхо и второй голос.
4. Попробуйте референс той же манеры и языка.
5. Только после этого подключайте RVC.

### Голос шумный или «металлический»

Причины обычно практические: плохой референс, слишком высокий Index, неудачная RVC-модель или чрезмерный Pitch shift. Сравните с RVC off и меняйте один параметр за раз.

### Index ничего не меняет

Проверьте наличие подходящего файла:

```text
models/rvc/<имя модели>.index
```

Без него `index_rate` не оказывает ожидаемого влияния.

### У локальной RVC-модели нет preview

Сам `.pth` не содержит аудиопример. Нужны сохранённые metadata или совпадение с записью каталога.

### RVC-модель скачалась, но не загружается

Если ошибка похожа на `invalid load key, '<'`, вместо весов могла сохраниться HTML-страница. Такие файлы должны отбрасываться проверкой checkpoint. Скачайте прямой `.pth`/`.zip` или откройте страницу через **🔗**.

### Генерация на CPU идёт долго

Это ожидаемо для XTTS, RVC и local LLM. Проверьте короткую фразу, закройте тяжёлые приложения или включите CUDA на совместимой NVIDIA GPU.

### Выбран GPU, но программа вернулась на CPU

Проверьте не только факт установки wheel, но и результат runtime-проверки. Torch требует `torch.cuda.is_available() == True`; llama.cpp GPU-backend должен реально открыть GGUF с `n_gpu_layers=-1`. Не прошедший вариант помечается broken, чтобы приложение не выбирало его снова автоматически.

### Установка Torch или llama.cpp оборвалась

Не удаляйте checkpoint вручную. Откройте настройки окружения: установщик может определить этап и предложить resume. Если пакет уже исправен, orphan-checkpoint будет очищен.

### Диагностика показывает, что RVC или llama.cpp не установлены

Это optional-компоненты. Их отсутствие не считается поломкой базового XTTS. Устанавливайте RVC только для voice conversion, а llama.cpp — только для локального AI.

### После очистки мусора что-то перестало импортироваться

Scanner сначала использует карантин и повторную диагностику. Не удаляйте `python/xtts_env/Quarantine` вручную: при новом сбое файлы должны вернуться автоматически. Если удаление уже подтверждено, используйте recovery и `logs/recovery_pip_output.log`.

### Облачный AI не отвечает

Проверьте API key, выбранную модель, доступность провайдера и VPN-требования. Для полностью локального режима выберите GGUF или отключите AI.

### После обновления произошёл откат

Updater сначала применяет файлы через staging и создаёт маркер подтверждения. Если следующий запуск не подтвердился, приложение возвращает backup. Диагностику смотрите в логах и окне системных настроек.

## Разработка и тесты

Конфигурация находится в `pyproject.toml`:

- Black и Ruff ориентируются на Python 3.10/3.11;
- Ruff проверяет E/F/W, а `F821` оставлен включённым для реальных undefined names;
- pytest использует корень проекта в `pythonpath`;
- `pytest-timeout` ограничивает один тест 60 секундами.

Запуск:

```text
test\run_tests.bat
```

или из подготовленного окружения:

```bash
pytest test
```

Тесты покрывают pipeline, chunker, normalizer, updater, environment setup, RVC, task manager, UI-модули и другие части проекта. Точное количество тестов меняется вместе с кодом, поэтому его лучше смотреть в актуальном CI.

При рефакторинге GUI важно сохранять публичные фасады. Например, `chat_settings.py` остаётся совместимым import/re-export слоем, хотя реализация настроек вынесена в несколько файлов.
