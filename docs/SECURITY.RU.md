# Политика безопасности

## Поддерживаемые версии

Обновления безопасности и исправления ошибок выпускаются для актуального опубликованного релиза XTTS Studio. Перед отправкой отчёта об ошибке убедитесь, что у вас установлена последняя версия приложения.

## Сообщение об уязвимости

**Не создавайте** публичные тикеты (GitHub Issues) для обнаруженных критических уязвимостей.

Используйте приватную форму сообщения об уязвимостях GitHub:  
`Security` → `Advisories` → `Report a vulnerability`

В отчёте укажите:
- затронутую версию и SHA256-хеш/коммит;
- версию Windows и конфигурацию CPU/GPU;
- минимальные шаги для воспроизведения проблемы;
- потенциальное влияние и необходимые условия для атаки;
- фрагменты логов с удалёнными API-ключами, личными текстами, референсами голосов и локальными путями;
- предполагаемый способ исправления, если известен.

Мейнтейнер обязуется подтвердить получение отчёта в течение 7 дней. Публичное раскрытие должно быть согласовано и выполнено только после выхода исправленного релиза.

## Границы безопасности

XTTS Studio обрабатывает голосовые референсы, созданное аудио и локальные модели с правами текущего пользователя Windows. Относитесь к сторонним файлам моделей `.pth`, `.index`, `.gguf`, `.zip` и `.whl` как к потенциально необработанному внешнему содержимому:

- Устанавливайте модели только из проверенных источников.
- Контрольная сумма SHA-256 подтверждает целостность файла относительно манифеста, но не гарантирует подлинность его издателя.
- Облачные AI-провайдеры получают текст, отправленный им пользователем. Локальный синтез XTTS/RVC не требует отправки текста или аудио сторонним сервисам.
- **Никогда** не прикрепляйте к публичным Issue файлы `json/gpt_settings.json`, истории генераций, записи голоса или сгенерированное приватное аудио.
- Исполняемый файл `XTTS Studio.exe` является удобной оболочкой-лаунчером. Он зафиксирован в Git (`!XTTS Studio.exe` в `.gitignore`), подписывается Ed25519-манифестом и обновляется автоапдейтером для защиты от рассинхронизации путей рантайма.

## Gate уязвимостей зависимостей (pip-audit)

CI запускает `tools/pip_audit_gate.py` поверх `requirements.txt`. `pip-audit` сообщает известные advisory, но не их severity, поэтому gate определяет severity каждой уязвимости из OSV и применяет политику:

- **Critical / High** → валят сборку, **если** advisory нет в `.security/pip-audit-allowlist.yml`;
- **Medium / Low** → warning (не валят);
- любая **просроченная** allowlist-запись → валят сборку.

`.security/pip-audit-allowlist.yml` — реестр задокументированных исключений. Каждая запись **обязана** содержать `id`, `package`, `reason`, `expires_at` (и `issue_link`). `expires_at` — триггер квартальной переоценки: по истечении CI снова краснеет, пока запись не продлят или advisory не закроют.

**Отклонение от «Critical всегда валит».** Замороженный ML-стек, от которого зависят XTTS v2 и RVC (`torch==2.2.2`, `transformers==4.38.2`), несёт Critical CVE без совместимого фикса в ближайшей перспективе — бамп torch/transformers ломает стек. Поэтому Critical **можно** подавить через allowlist, но **только** как явное, датированное, обоснованное исключение, никогда молчаливо. Новая Critical CVE, которой нет в allowlist, по-прежнему валит сборку. Исключения зафиксированы per-advisory с конкретной причиной, почему уязвимый code path недостижим или смягчён в этом проекте (модель доверия `torch.load`, неиспользуемые классы моделей, только eager-inference), и продублированы в [SECURITY_BASELINE](./SECURITY_BASELINE.RU.md).

## Требования к безопасности релизов

Официальный публичный релиз приложения должен соответствовать следующим критериям:

1. Наличие подписанного манифеста обновления (`json/version.json.sig`), валидируемого Ed25519-ключом до выполнения обновления.
2. Безопасная распаковка архивов обновлений с зафиксированной защитой от Zip-Slip и контролем путей в `_update_staging`.
3. Публикация контрольных сумм SHA-256 для всех файлов в `checksums.txt`.
4. Сгенерированный детерминированный SBOM (`json/sbom.cdx.json`).
5. `pip-audit` запускается в CI как blocking gate по High/Critical CVE с явным allowlist (см. раздел «Gate уязвимостей зависимостей»).
6. Отсутствие открытых незашифрованных API-ключей в файлах конфигурации.
7. Явное подтверждение доверия при загрузке невалидированных чеклистов/моделей PyTorch.
